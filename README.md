# archon

Lightweight dependency graph and AI context generator for multi-repo ecosystems.

## What is archon?

archon is a developer tool that provides architecture-level intelligence for multi-repo Rust projects. It sits alongside the repos it governs and answers questions that individual repos can't: which repos exist, how they connect, what public APIs they expose, and what contracts they implement.

In a multi-repo ecosystem, a developer working in one repo doesn't naturally know what types another repo exports or which downstream repos will break when they change a public API. archon solves this by:

1. **Extracting** each repo's public API surface via Rust AST analysis
2. **Assembling** a dependency graph from per-repo manifests
3. **Generating** AI context that gets injected into each repo's `CLAUDE.md`, giving agents full ecosystem awareness
4. **Enforcing** rules and contracts across repos
5. **Describing** relationships in natural language — tell archon how repos connect in plain English and it updates manifests automatically

archon is a **tool**, not a library. No other repo depends on it at runtime. It reads from sibling repos and writes context back to them.

## Install

```bash
cargo install --path .
```

Requires Rust 2021 edition. AI features (`init`, `describe`, `assemble --bootstrap`) require the [`claude` CLI](https://claude.ai/claude-code) but all core graph operations work without it.

## Quick start

```bash
# 1. Initialize a manifest in your repo
cd ~/work/my-project
archon init

# 2. Extract your public API surface
archon scan

# 3. Assemble the ecosystem graph and distribute context
cd ~/work/architecture-repo
archon assemble --root .. --registry archon-registry --distribute

# 4. Validate
archon check
archon verify
```

## Core concepts

### Manifest (`archon.yaml`)

Each repo has a small manifest declaring its identity, dependencies, and rules:

```yaml
name: hologram-sandbox
description: "Sandboxed execution environment for hologram programs"
role: extension
depends_on:
  - hologram
  - hologram-ai
provides:
  - sandbox-runtime
crates:
  - name: hologram-sandbox
    public: true
registry: ../hologram-architecture/archon-registry
rules:
  - id: tests-pass
    run: "cargo test --all"
    description: "All tests pass across the workspace"
  - id: no-clippy-warnings
    run: "cargo clippy --all -- -D warnings"
    description: "No clippy warnings or errors"
  - id: builds-clean
    run: "cargo build --all"
    description: "All crates build successfully"
```

| Field | Required | Description |
|-------|----------|-------------|
| `name` | yes | Repo identifier |
| `description` | yes | What the repo does |
| `role` | yes | One of: `core`, `extension`, `tool`, `service`, `library` |
| `depends_on` | no | List of dependency repo names |
| `provides` | no | List of capabilities/features this repo provides |
| `crates` | no | Workspace crate entries with `name` and `public` flag |
| `registry` | no | Path to archon registry for auto-discovery |
| `rules` | no | Conformance rules with shell commands |

**Roles:**
- **core** — central system component (runtime, compiler, engine)
- **extension** — extends or integrates with a core repo
- **tool** — developer tool or CLI utility
- **service** — deployed service or API
- **library** — shared library consumed by other repos

### Registry

A registry is a directory (typically in a dedicated architecture repo) that collects broadcasts and the assembled graph:

```
archon-registry/
  graph.yaml                    # assembled dependency graph
  broadcasts/
    hologram.yaml               # per-repo API surface
    hologram-sandbox.yaml
  context/
    hologram.context.md         # per-repo AI context
    hologram-sandbox.context.md
```

### Contract annotations

Tag public items with contracts using doc comments:

```rust
/// @contract(execution-plan)
pub struct ExecutionPlan { ... }

/// @contract(execution-plan)
pub fn compile(graph: &Graph) -> Result<ExecutionPlan> { ... }
```

`archon scan` picks these up and includes them in the broadcast, distinguishing stable contract APIs from incidental public items.

### AI context injection

archon generates context blocks injected into `CLAUDE.md` between managed markers:

```markdown
<!-- ARCHON:CONTEXT:BEGIN -->
## Ecosystem Context (auto-generated by archon)

See [`.archon/context.md`](.archon/context.md) for full dependency graph,
public API surface, and contract details for this repo.
<!-- ARCHON:CONTEXT:END -->
```

The full context in `.archon/context.md` tells AI agents:
- **Where they are** — the repo's name, role, and description
- **What they depend on** — upstream repos with key types, traits, and functions
- **Who depends on them** — downstream repos consuming their public API
- **Their public API surface** — types, traits, and functions they expose, including contract membership

If a repo doesn't have a `CLAUDE.md` when context is distributed, archon creates one automatically with the managed markers.

## Commands

### `archon init`

Create an `archon.yaml` manifest for a repo. Runs an interactive wizard that auto-detects Cargo.toml workspace members and generates default rules.

```bash
archon init                                        # interactive wizard
archon init --registry ../archon-registry          # discover sibling repos for depends_on
archon init --owner "@my-team"                     # set repo owner
archon init --no-ai                                # skip AI inference for role/description
```

| Flag | Description |
|------|-------------|
| `--path <PATH>` | Directory to initialize (default: `.`) |
| `--registry <PATH>` | Registry directory for contract discovery |
| `--owner <STRING>` | Repo owner (default: `@team`) |
| `--no-ai` | Skip AI inference even if `claude` CLI is available |

Non-interactive mode activates automatically when stdin is not a TTY (useful for CI).

### `archon scan`

Extract the public API surface of your crates using `syn`-based AST analysis. Writes `.archon/broadcast.yaml` locally. If a registry is accessible, copies the broadcast there and generates `.archon/context.md`.

```bash
archon scan                                        # uses registry from archon.yaml if set
archon scan --registry ../archon-registry          # explicit registry path
```

| Flag | Description |
|------|-------------|
| `--path <PATH>` | Directory containing `archon.yaml` (default: `.`) |
| `--registry <PATH>` | Path to registry directory |

### `archon assemble`

Collect all `archon.yaml` manifests from sibling directories, build `graph.yaml`, generate per-repo AI context, and optionally distribute context to each repo.

```bash
archon assemble --root .. --distribute
archon assemble --root .. --registry archon-registry --distribute
archon assemble --root .. --bootstrap --distribute    # auto-create manifests for new repos
archon assemble --root .. --bootstrap --no-ai         # bootstrap without AI inference
```

| Flag | Description |
|------|-------------|
| `--root <PATH>` | Root directory containing repo directories (default: `..`) |
| `--registry <PATH>` | Path to registry directory (default: `archon-registry`) |
| `--distribute` | Inject context into each repo's `CLAUDE.md` and `AGENTS.md` |
| `--bootstrap` | Auto-create `archon.yaml` for repos with `Cargo.toml` but no manifest |
| `--no-ai` | Skip AI inference during bootstrap |

With `--distribute`, archon:
1. Writes `.archon/context.md` in each repo (always)
2. Replaces the managed section in `CLAUDE.md`/`AGENTS.md` with a reference to the context file
3. Creates `CLAUDE.md` with markers if neither file exists

### `archon check`

Validate graph consistency across the ecosystem. Reports missing dependencies, dependency cycles, and manifest parse errors.

```bash
archon check                     # assemble from sibling repos and check
archon check --format json       # machine-readable output for CI
archon check --graph graph.yaml  # check a pre-built graph file
```

| Flag | Description |
|------|-------------|
| `--graph <PATH>` | Path to `graph.yaml` or registry directory |
| `--root <PATH>` | Root directory to assemble from (default: `..`) |
| `--format <FORMAT>` | Output format: `text` or `json` (default: `text`) |

### `archon verify`

Run the conformance rules defined in `archon.yaml` for the current repo.

```bash
archon verify
# rule: All tests pass across the workspace ... pass
# rule: No clippy warnings or errors ... pass
#
# ok: All 2 rules passed for hologram-sandbox
```

| Flag | Description |
|------|-------------|
| `--path <PATH>` | Directory containing `archon.yaml` (default: `.`) |
| `--format <FORMAT>` | Output format: `text` or `json` (default: `text`) |

### `archon describe`

Describe how repos connect in natural language and update manifests automatically. Requires the `claude` CLI.

```bash
# Set up relationships
archon describe "hologram is the core runtime. hologram-ai extends it with AI inference."

# Add a dependency
archon describe "hologram-sandbox also depends on hologram-ai"

# Preview changes first
archon describe --dry-run "archon is a developer tool, not a library"

# Debug AI response
archon describe --verbose "hologram-network depends on hologram and UOR-Framework"
```

| Flag | Description |
|------|-------------|
| `--root <PATH>` | Root directory containing repo directories (default: `..`) |
| `--dry-run` | Preview changes without writing |
| `--verbose` | Show raw AI response for debugging |

The command reads all existing manifests, sends your description to Claude, and applies inferred changes (role, description, depends_on, provides) to each repo's `archon.yaml`. If the described state already matches the current configuration, archon reports which repos are already up to date rather than a generic "no changes" message.

### `archon dashboard`

Interactive architecture dashboard for exploring the dependency graph.

```bash
archon dashboard --root ..                          # TUI mode (interactive terminal)
archon dashboard --root .. --web                    # generate static HTML dashboard
```

| Flag | Description |
|------|-------------|
| `--root <PATH>` | Root directory containing repo directories (default: `..`) |
| `--registry <PATH>` | Path to registry directory (default: `archon-registry`) |
| `--web` | Generate `archon-dashboard.html` instead of launching TUI |

### `archon graph`

Query and visualize the dependency graph. All subcommands support `--graph` to use a pre-built `graph.yaml` or `--root` to auto-assemble from manifests.

#### `archon graph show`

Display the full dependency graph as a table.

```bash
archon graph show                     # all repos
archon graph show --role core         # filter by role
archon graph show --format json       # JSON output
```

| Flag | Description |
|------|-------------|
| `--role <ROLE>` | Filter by role: `core`, `extension`, `tool`, `service`, `library` |
| `--format <FORMAT>` | Output format: `text` or `json` (default: `text`) |

#### `archon graph info <repo>`

Show detailed information about a repo — dependencies, dependents, contracts, and transitive impact.

```bash
archon graph info hologram-sandbox
```

#### `archon graph deps <repo>`

Show what a repo depends on.

```bash
archon graph deps hologram-sandbox            # transitive (default)
archon graph deps hologram-sandbox --direct   # direct only
```

#### `archon graph rdeps <repo>`

Show what depends on a repo (impact analysis).

```bash
archon graph rdeps hologram              # who breaks if hologram changes?
archon graph rdeps hologram --direct     # direct dependents only
```

#### `archon graph path <from> <to>`

Find the shortest dependency path between two repos.

```bash
archon graph path my-tool hologram
# dependency path:
#   my-tool
#     ↓ depends on
#   hologram-sandbox
#     ↓ depends on
#   hologram
```

## Architecture repo workflow

An architecture repo is a dedicated repo that holds the assembled graph and collected broadcasts for the entire ecosystem:

```
~/work/
  hologram/                 # core repo (has archon.yaml)
  hologram-ai/              # extension repo
  hologram-sandbox/         # extension repo
  hologram-architecture/    # architecture repo (registry lives here)
```

### Initial setup

```bash
# Create the architecture repo
mkdir hologram-architecture && cd hologram-architecture
git init
mkdir -p archon-registry/broadcasts

# Bootstrap — auto-create manifests for all sibling repos
archon assemble --root .. --bootstrap --registry archon-registry --distribute

# Configure relationships in plain English
archon describe --root .. "hologram is the core runtime. hologram-ai extends it \
  with AI inference. hologram-sandbox provides sandboxed execution depending on hologram."

# Re-assemble with updated manifests
archon assemble --root .. --registry archon-registry --distribute
archon check
```

### Adding a new repo

```bash
# From the new repo:
cd ~/work/new-repo
archon init --registry ../hologram-architecture/archon-registry
archon scan --registry ../hologram-architecture/archon-registry

# From the architecture repo:
cd ~/work/hologram-architecture
archon assemble --root .. --registry archon-registry --distribute
```

### Ongoing maintenance

```bash
# After API changes in any repo, re-scan and re-assemble:
cd ~/work/hologram
archon scan

cd ~/work/hologram-architecture
archon assemble --root .. --registry archon-registry --distribute
archon check
git add -A && git commit -m "update graph"
```

## CI integration

```yaml
# .github/workflows/archon.yml
name: archon
on:
  push:
    paths: ['archon.yaml', 'archon-registry/**']
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cargo install --path path/to/archon
      - run: archon check --format json
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: archon verify --format json
```

## How AI agents use archon context

When an AI agent lands in a repo that participates in the archon ecosystem, it finds a managed section in `CLAUDE.md` pointing to `.archon/context.md`. This file tells the agent:

- **Where it is** — the repo's name, role, and description
- **What it depends on** — upstream repos with their key types, traits, and functions
- **Who depends on it** — downstream repos that consume its public API
- **Its public API surface** — types, traits, and functions it exposes, including contract membership

This means agents don't need to guess how a repo connects to the rest of the ecosystem. If an agent is working in `hologram-sandbox` and needs to call something from `hologram`, the context lists the exact types and function signatures available. If it's about to change a public API, the context tells it which downstream repos will break.

## Source code

| Module | Purpose |
|--------|---------|
| `src/main.rs` | CLI definition and all command handlers |
| `src/manifest.rs` | `archon.yaml` data model and serialization |
| `src/extract.rs` | Rust AST parsing with `syn` to extract public API surface |
| `src/broadcast.rs` | Per-repo API surface data model |
| `src/graph.rs` | Dependency graph assembly, cycle detection, violation checking |
| `src/context.rs` | AI context markdown generation from graph + broadcasts |
| `src/inject.rs` | Managed section injection into `CLAUDE.md` / `AGENTS.md` |
| `src/render.rs` | Output rendering utilities |
| `src/dashboard/` | Interactive TUI and static HTML dashboard |
